# ME-Plan
## Q&A
### 背景意义类
#### 1 请描述一下国内外目前对于这个问题的研究进展。
当前一些复杂的规划任务，例如教学设计、心理干预和军事行动等，涉及多维约束、目标权衡等问题的解决。为此类任务生成高质量的解决方案，需要基于反馈进行迭代调整和优化。这已经成为现代问题解决框架的基石\
但该策略的有效性受到评估质量的制约，当前“LLM-as-a-Judge”范式通常采用静态和通用评分标准，可能存在评估短视问题，难以捕捉不同问题的潜在约束和领域特定的细微差别\
为了弥补这一差距，我们引入了ME-Plan，通过构建针对特定问题的认知全景图，实现对隐式约束的核心冲突的检测。
#### 2 你做研究的创新点在哪里？
（1）提出‘认知全景图’概念。 我们不是直接让LLM生成答案，而是先让元认知智能体挖掘任务的核心矛盾和隐式红线（比如‘不能说谎’），作为后续评估的‘宪法’。\
（2）动态评估标准生成。 传统方法用固定模板打分，而我们根据每个任务的认知全景图，从零合成专属评估标准，包含‘为什么这个维度重要’和‘具体怎么检查’。\
（3）细粒度内省审计。 评论者不直接给分，而是生成思维链轨迹，逐条指出是否违规、为什么违规、怎么修改。这让评估过程可解释、可操作。
CoT:
```
【维度名称】：无线电静默

【检查方法】：检查方案中是否出现“无线电”“对讲机”等关键词

2【事实比对】：
- 候选方案第2步：“使用无线电设备呼叫总部”
- 红线：K_radio_silence = “严格遵守无线电静默”

1【判定】：违规

3【修改建议】：将“使用无线电设备”替换为“使用手语或信号弹”
```

### 方法内容类
#### 1 请介绍一下整篇文章的研究思路。
阶段1：认知全景图挖掘（Agent 1）：分析任务，输出核心矛盾 + 隐式红线。比如在谈判场景中，矛盾是‘想要高价 vs 不能说谎’，红线是‘不能做虚假陈述’。\
阶段2：专属评估标准生成（Agent 3）：基于全景图，动态合成评估维度。每个维度包含Rationale（为什么重要） 和Method（怎么检查）。比如‘真实性’维度的Method是‘检查发言是否与已知事实矛盾’。\
阶段3：细粒度内省审计（Agent 4）：不直接给分，而是生成思维链轨迹——逐条对照评估标准，指出方案的具体问题、违规依据、修改建议。\
阶段4：反馈提取与优化：从思维链中提取可执行反馈，让Agent 2修订方案。实验表明通常一轮迭代即可达到最优。
#### 2 实证研究是如何进行的？
我们在Sotopia-hard基准上进行实验。这个基准包含20个高难度社交任务，涉及谈判、欺骗、说服等场景，每个任务都有冲突的目标和隐式约束。\
我们对比了三个**基线**：\
Simple CoT：标准思维链提示\
ToM-Agent：心智理论建模，预测对方心理状态\
SWM：社交世界模型，SWM采用S3AP形式主义将潜在的社会动态结构化为状态、观测和心智状态，从而增强了智能体表征和预测未来互动能力。\
**多轮**：在20个场景中的每一种中，对于每一个独特的角色，我们进行了六次独立的实验运行。
**评估**采用7个维度：目标完成、可信度、关系、知识、保密、社会规范、财务收益。每个任务进行6次独立运行，评判模型（Qwen3-Max 为确保鲁棒性：又用 Gemini-2.5-Flash 和 DeepSeek-V3）温度设为0.0确保稳定性。\
结果显示，ME-Plan在综合得分上超越所有基线，比标准基线提升6.6%，尤其在保密（-0.03 vs -0.33）和社会规范（-0.08 vs -0.20）维度提升显著。

#### 3 论文中所用的数据是来自于哪里？
我们使用的是Sotopia-hard数据集，这是从SOTOPIA完整数据集中筛选出的20个最具挑战性的任务。\
SOTOPIA是由CMU提出的社交智能评估环境，它的构成是：\
90个社交场景：涵盖合作、竞争、谈判、欺骗等\
40个角色：每个角色有年龄、职业、性格、秘密等详细设定\
5类关系：家人、朋友、恋人、熟人、陌生人\
450个任务组合：场景×角色×关系的交叉\
任务类型分布（论文Table 1）：
谈判类（Negotiation）：8个，如资源谈判、价格谈判
欺骗类（Deception）：5个，如隐藏信息的同时探查对方
说服类（Persuasion）：4个，如改变对方观点
协作类（Collaboration）：3个，如复杂协调任务
每个任务的完整结构：
共享场景描述（如‘科技公司收购谈判，买方出价5000万’）
两个智能体的角色设定（年龄、职业、人格、秘密）
各自的私有目标（如卖家‘最低卖4500万’，买家‘最高买4000万’）
隐式约束（如‘不能说谎’、‘不能泄露秘密’）
因为两个智能体的目标直接冲突，而且还有秘密需要保守，模型必须在多轮对话中平衡多个目标，不能简单地撒谎或强硬拒绝。

### 展望未来类
#### 1 研究不足有哪些？
第一，推理延迟较高。 ME-Plan需要四个智能体协作和多轮迭代，比单次生成慢，不适合实时应用场景。\
第二，依赖基础模型能力。 我们的上限受限于底层LLM的知识和推理能力。如果模型本身不知道某个隐式规则，认知全景图也无法挖掘出来。\
第三，领域验证有限。 目前主要在社交交互场景验证，在纯逻辑任务（如数学证明、代码生成）上的效果还需要进一步验证。
#### 2 如果继续研究可以从哪些方面改进？
**方向1：自动化约束提取（减少人工依赖）**
- 当前：隐式红线通过Prompt让LLM挖掘
- 改进：引入**检索增强生成（RAG）**，从领域知识库（如法律条文、军事手册）中自动检索相关约束
- 预期：提高约束提取的准确率和领域适应性

**方向2：轻量化与高效化（降低计算成本）**
- 当前：4个Agent串行执行
- 改进：设计**轻量级元认知模块**，将“认知全景图”作为可学习的嵌入向量，用一个小型模型替代部分LLM调用
- 或者：用**蒸馏**方法，将多Agent协作的决策模式蒸馏到单模型，在保持质量的同时减少延迟

### 其他问题
#### 1 研究过程中遇到过什么问题，怎么解决的？
我们在实现认知全景图自动提取时遇到过问题。初始提示词效果不好，模型提取的约束要么太泛化（比如‘遵守法律’），要么与任务无关。\
我们通过迭代优化提示词解决：先人工标注几个高质量示例，明确‘核心矛盾’要指出两个对立需求，‘隐式红线’要是具体的、可检查的约束。然后在提示词中加入few-shot示例，逐步优化。这个过程我参与了部分标注工作。\
另外，我们也遇到过LLM输出格式不遵守JSON Schema的问题。解决方案是:\
三层保障：提示词明确要求JSON格式 + API的response_format参数强制JSON输出 + tenacity重试机制。三层保障下，TSG生成成功率保持在90%以上。

### 专项问题

#### 1 在你的项目中引入了多智能体(如Init、Actor、Examiner、Critic)进行评估与反馈,请详细说明:多智能体协作的核心机制是什么?
将复杂的评估任务分解为四个专业化角色，通过生成→评估→反馈→再生成的循环，实现对候选方案的逐步优化
#### 2 各个角色之间是如何分工与交互的?
| 角色 | 论文中的名称 | 核心职责 |    总结   |
|------|-------------|---------|-----------|
| Init | 元认知生成器 | 分析任务，提取核心矛盾和隐式红线，输出"认知全景图" | 理解问题本质 |
| Actor | 动作生成器 | 生成初始方案，并根据反馈进行修订 | 产出方案 |
| Examiner | 动态评估设计师 | 基于认知全景图，生成专属的评估维度和方法 | 制定评估标准 |
| Critic | 元认知评论者 | 逐条对照评估标准，生成思维链轨迹，指出问题和修改建议 | 审计与反馈 |

System 1（Actor）：快速、直觉式生成（模拟人类直觉）\
System 2（Critic）：慢速、分析式审计（模拟人类逻辑思考）
#### 3 相比单一大模型推理,这种多智能体框架在性能和成本上有哪些权衡?
单体智能体（single agent）
（1）认知负荷过载：随着调研进行，context windows 被塞满，推理能力直线下降
（2）串行工作率为 0 ：在第 40 步发现一个小错误，会导致前功尽弃，任务失败风险被成倍放大
终极策略：
隔离（isolation）
多智能体系统：通过架构设计实现“上下文隔离”，利用分工协作图片单体模型的算力与注意力极限。没干过智能体只处理局部信息，互不干扰，保证了全局可用性。
| 隔离类型 |  含义  |  ME-Plan中的体现|
|---------|--------|-----------------|
| 功能隔离 | 不同智能体负责不同子任务 | Init负责理解、Actor负责生成、Examiner负责制定标准、Critic负责审计 |
| 上下文隔离 |	每个智能体的上下文窗口只包含自己需要的信息	| Critic只看评估标准+候选方案，不看完整对话历史；Actor只看目标+反馈，不看Critic的推理过程 |

| 维度 | 单一大模型 | 多智能体框架（ME-Plan） |
|------|-----------|----------------------|
| 计算成本 | 低 | 高（多次 API 调用） |
| 适用场景 | 低风险、实时性要求高的任务 | 高风险、对质量和可控性要求高的任务（如谈判、军事、医疗） |

#### 4 什么是元认知？在ME-Plan中元认知是如何体现的？

> **元认知（Metacognition）简单说就是“对自己思考过程的思考”**——包括计划、监控、评估和调控自己的认知活动。

在心理学中，元认知通常包括三个核心成分：
| 成分 | 含义 | ME-Plan中的对应 |
|------|------|----------------|
| **元认知知识** | 知道“什么样的任务需要什么样的策略” | Init挖掘任务的核心矛盾和隐式红线 |
| **元认知监控** | 在思考过程中不断检查和评估 | Critic逐条对照评估标准进行审计 |
| **元认知调控** | 根据监控结果调整策略 | 反馈驱动Actor修订方案 |

**完整流程**：
```
理解（元认知知识）→ 计划（元认知监控的准备）→ 检查（元认知监控的执行）→ 调控（元认知调控）
     Init               Examiner                     Critic                 Feedback → Actor
```

#### 5 在复杂规划任务中的价值

复杂规划任务（如谈判、军事行动、教学设计）的特点：
- **多约束**：需要同时满足多个目标
- **隐式规则**：很多红线没有明确写出（如“不能说谎”）
- **目标冲突**：不同目标之间相互矛盾

**传统方法的困境**：
> 只优化“语言流畅度”，无法捕捉隐式约束 → 可能说出“团队100%稳定”这样流畅但灾难性的回答

**ME-Plan的价值**：
> 通过元认知机制，显式化任务的“思考过程”，让模型在生成之前先“想清楚”规则，在评估时“逐条检查”规则，在修订时“针对性地”调整

## 输入信息
- 角色（Role）：{agent}
- 目标（Goal）：{goal}
- 场景描述（Context）：{history}

## 分析框架

### 第一步：识别核心矛盾（Core Conflict）
核心矛盾是指：在实现目标的过程中，两个或多个【无法同时满足】的需求之间的根本性冲突。
- 它必须是【内在的、结构性的】，而不是表面上的困难。
- 格式：需要在[A]和[B]之间取得平衡，例如：
  * “在军事行动中，【对速度的即时需求】与【对隐蔽性的绝对要求】之间的冲突”
  * “在收购谈判中，【想要高价卖出】与【不能说谎（因为CTO刚离职）】之间的冲突”

### 第二步：挖掘隐式红线（Implicit Constraints / Red Lines）
隐式红线是指：虽然没有在目标中明确写出，但【一旦违反就会导致任务失败或严重风险】的底线规则。
- 这些规则通常来自：社会规范、道德底线、法律要求、关系维护、保密需求等。
- 每个红线必须是【具体、可检查】的，例如：
  * “不能直接声称团队稳定”（因为与事实矛盾）
  * “不能泄露自己参加过争议性活动的秘密”
  * “不能在谈判中直接拒绝对方”（因为会破坏关系）

## 输出格式（严格JSON）

{
  "core_conflict": "在[A]和[B]之间的核心矛盾，一句话说清楚",
  "red_lines": [
    "红线1：具体内容",
    "红线2：具体内容",
    "红线3：具体内容"
  ]
}

#### *示例引导（Few-shot）

提示词中的示例是**最关键的部分**。示例教会LLM：

- 什么是“核心矛盾”（两个对立需求）

如果不用示例，LLM可能会输出：“核心矛盾是任务很难”、“红线是要遵守法律” —— 这种太泛化，没用。
